# Open Secure Tunneling Protocol (OSTP) - Core Issues

После анализа реализации OSTP (на основе UDP и Noise Protocol Framework), были выявлены следующие **архитектурные и криптографические проблемы**, требующие немедленного решения перед использованием протокола в продакшене.

## 1. Отсутствие механизма Anti-Replay (Уязвимость к атакам повторного воспроизведения)

В текущей реализации `ProtocolMachine` (`ostp-core/src/protocol.rs`) при обработке входящих пакетов извлекается `nonce` в открытом виде (8 байт), после чего пакет дешифруется с использованием `SessionCipher`:
```rust
let nonce = u64::from_be_bytes(raw[4..12].try_into().unwrap());
let plaintext = cipher.decrypt(nonce, ciphertext, &session_id_bytes)?;
```
**Проблема:**
В коде присутствует поле `_recv_nonce: u64`, но оно не обновляется и никак не проверяется. Так как UDP протокол не имеет встроенной защиты от дублирования, злоумышленник может перехватить любой легитимный датаграмм (например, команду на создание TCP-соединения) и бесконечно отправлять его повторно на сервер. Сервер успешно дешифрует каждый из них (так как nonce и ключ совпадают) и будет плодить "фантомные" действия.

**Решение:**
Необходимо реализовать "скользящее окно" (Sliding Window / Anti-Replay Window) с битовой маской, чтобы отбрасывать `nonce`, которые выходят за рамки окна или уже были получены.

## 2. Отсутствие надежности и порядка (Reliability & Ordering)

OSTP туннелирует потоки TCP (внутри `RelayMessage::Data`), разбивая их на сырые UDP-датаграммы:
```rust
match remote.try_read(&mut buf) {
    Ok(n) => send_relay_to_stream(..., RelayMessage::Data(buf[..n]), ...).await;
}
```
**Проблема:**
Если любой из UDP-пакетов потеряется в сети, часть TCP-потока исчезнет безвозвратно. Если пакеты придут в неверном порядке, данные смешаются. OSTP не имеет механизмов подтверждения (ACK), управления окном, нумерации сиквенсов (Sequence Number) и повторной передачи потерянных пакетов (ARQ). Это полностью нарушает контракт TCP для конечного приложения — скачанные файлы будут битыми, а HTTP-запросы будут падать.

**Решение:**
Необходимо либо перевести транспорт на TCP, либо (если цель — использование UDP) интегрировать надежный транспортный уровень поверх UDP (например, KCP, QUIC, или собственный ARQ/Sliding Window механизм с ACK и Retransmission).

## 3. Отсутствие контроля перегрузки сети (Congestion Control)

**Проблема:**
Сервер читает из TCP-сокета (`try_read`) и немедленно выплевывает данные в UDP-сокет без какого-либо сдерживания. Если канал клиента не может пропустить такой объем (например, медленный 3G), UDP пакеты будут массово дропаться на промежуточных узлах. 

**Решение:**
Необходим алгоритм управления перегрузкой (например, BBR или Cubic), который будет управлять окном отправки, либо использование готовых решений вроде QUIC.
