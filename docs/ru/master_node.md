# Мастер-нода и инфраструктура прокси

Мастер-нода является центральным связующим звеном и координатором трафика в сети ByteAway. Она выполняет функции учета, балансировки нагрузки для B2B-клиентов и оркестрации распределенной сети мобильных узлов.

## Архитектура и технологии

Серверная часть разработана на **Rust** с использованием асинхронной среды выполнения **Tokio** и фреймворка **Axum**. Проект использует СУБД **PostgreSQL** для хранения постоянных данных и **Redis** для кэширования и быстрого обмена информацией.

---

## Основные подсистемы

### 1. REST API (`/api/v1`)
Обеспечивает взаимодействие с мобильными приложениями, B2B-клиентами и панелями администратора:
- **Авторизация**: Реализована аутентификация по токенам. Проверяется статус учетных записей и их доступность в сети.
- **Управление прокси**: Генерация конфигураций для подключения к VPN и получение текущих сетевых лимитов.
- **Биллинг**: Пополнение баланса, расчет стоимости трафика и хранение истории транзакций.

### 2. Мультиплексированные туннели
Мобильные узлы подключаются к Мастер-ноде через выделенные ингрессы. Система поддерживает несколько протоколов связи:
- **QUIC**: Основной протокол на базе UDP. Обеспечивает высокую скорость и устойчив к смене типа подключения на мобильном устройстве (например, при переходе с LTE на Wi-Fi).
- **WebSocket (WS)**: Резервный протокол. Используется для обхода строгих сетевых ограничений и брандмауэров, пропускающих только стандартный HTTP/HTTPS трафик.
- **Hysteria2**: Альтернативный UDP-протокол, использующий агрессивный алгоритм контроля перегрузок для максимизации пропускной способности на нестабильных каналах связи.

Поверх физических соединений запускается мультиплексор **Yamux**. Это позволяет передавать тысячи независимых сетевых сессий внутри одного установленного соединения.

### 3. SOCKS5/HTTP Гейтвей
Прокси-сервер, который принимает входящие запросы от B2B-клиентов и перенаправляет их на мобильные устройства.
- При получении нового TCP-соединения гейтвей запрашивает у реестра доступный мобильный узел.
- Запрос упаковывается в yamux-канал и отправляется на выбранный мобильный узел.
- Мобильный узел производит запрос в интернет и возвращает ответ через Мастер-ноду конечному B2B-клиенту.

### 4. Движок Биллинга (Billing Engine)
Отвечает за расчет использованного трафика и списание средств:
- Вся информация о пропущенном трафике (объем в байтах) сначала аккумулируется в оперативной памяти и кэше Redis.
- Каждые 60 секунд фоновый воркер производит сброс накопленных данных (Flush) из Redis в основную СУБД PostgreSQL.
- На основе объема переданных данных с баланса B2B-клиента списывается сумма, соответствующая тарифной сетке (по умолчанию фиксированная цена за 1 ГБ).

## Управление состоянием (Registry)

Для быстрого поиска активных узлов система использует Redis-реестр `RedisNodeRegistry`:
- При подключении мобильного узла создается запись с уникальным идентификатором устройства и его характеристиками (страна, регион, скорость соединения).
- Записи имеют ограниченный срок действия (TTL) и автоматически удаляются, если мобильный узел перестал отправлять пинг-запросы.
- Балансировщик нагрузки выбирает узлы из пула на основе текущей загрузки канала и географических требований B2B-клиента.
